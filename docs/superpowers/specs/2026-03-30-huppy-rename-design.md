# Huppy Rename Migration Design

**日期：** 2026-03-30
**版本：** v1.0
**状态：** 待审核

---

## 一、目标

将当前项目从 **Hoppy / hoppy** 统一迁移为 **Huppy / huppy**，并以 `huppy.ai` 作为正式品牌域名。

这次迁移的目标不是做一次机械全局替换，而是在保证产品**继续可用、可构建、可上线**的前提下，完成品牌、命令名、配置名、文档和默认服务入口的统一。

本次设计遵循以下原则：

- 对用户可见品牌全部统一为 `Huppy`
- 对直接面向用户的命令名统一为 `huppy`
- 尽量保留 Happy 原始协议名、上游引用、版权说明和内部历史语义
- 不为了“名字彻底干净”而破坏现有可用性
- 迁移按层推进，每层都要能单独验证

---

## 二、为什么不能直接粗暴全局替换

项目里当前同时存在三类名称：

1. **上游来源名称**
   - `Happy`
   - 用于版权、协议文档、上游来源说明、兼容历史语义

2. **现品牌名称**
   - `Hoppy / hoppy`
   - 已进入 App 配置、UI 文案、设计文档、图标、CLI 标题、部分包名和目录语义

3. **目标品牌名称**
   - `Huppy / huppy`

如果现在直接做不区分语义的字符串替换，会产生几个高风险问题：

- 把应保留的 `Happy` 上游引用误改掉
- 把协议语义里的 `Happy server`、`happy-agent`、`happy-cli` 直接替掉，导致兼容性和说明文档混乱
- 把目录、包、命令、环境变量、显示文案混为一谈，导致构建或运行失败
- 让正在规划中的 M3 后端迁移基线变得不清晰

因此这次迁移必须按“语义分层”实施，而不是一次性 `search/replace`。

---

## 三、迁移范围

### 必须改

- App 显示名
- 登录页、欢迎页、设置页、菜单栏等用户可见品牌文案
- 官网域名、深链域名、默认后端域名中的品牌命名
- CLI 对用户展示的命令名称和帮助文案
- 文档中的当前品牌介绍、截图、设计稿标题
- 商店元数据、描述、订阅名称、品牌 Slogan
- 图标文件、logo 资源、导航栏品牌图形

### 计划改

- 包名中显式出现 `hoppy` 的位置
- 配置文件中的品牌 slug
- 默认环境变量名和默认 endpoint 标识
- README、handoff、计划文档、spec 文档中的 Hoppy 品牌表述

### 不应机械替换

- `Happy` 上游版权与来源说明
- 描述原始 Happy 协议、Happy Server、Happy CLI 行为的历史文档
- 明确是在讨论上游项目时出现的 `Happy`
- 与兼容层直接绑定、短期内不值得动的内部代码符号

---

## 四、推荐策略

推荐采用 **“完整品牌统一目标 + 分层安全迁移执行”**。

也就是说，最终目标是：

- 用户看到的是 `Huppy`
- 用户输入的命令是 `huppy`
- 主要品牌资源、文档、配置默认值都改成 `huppy`

但执行方式不是一把梭，而是按下面四层推进：

### Layer 1：对外品牌层

先统一所有用户直接看到的品牌：

- App 名称
- 标题栏
- 登录页/欢迎页
- 图标与 logo
- 翻译文案
- 商店/官网/说明页品牌名

这一层的目标是让用户看到的世界已经完全是 `Huppy`。

### Layer 2：开发者入口层

再统一开发者直接接触的入口：

- CLI 命令名
- 帮助文案
- README 中的启动命令
- 安装说明
- 默认配置示例

这一层要重点控制兼容性。必要时允许保留短期别名，例如在内部实现上兼容旧命令入口，但对外只宣传 `huppy`。

### Layer 3：配置与默认值层

再统一配置和默认资源命名：

- `app.config.js`
- `package.json` 中的名称/slug/displayName
- Tauri/Expo/桌面端配置
- 默认 API endpoint
- 默认域名
- 深链、二维码、邀请链接等带品牌标识的入口

这一层的目标是让新安装和新构建产物天然使用 `huppy`。

### Layer 4：文档与计划层

最后统一项目内品牌文档：

- `docs/superpowers/specs`
- `docs/superpowers/plans`
- `HANDOFF.md`
- mockups 标题
- README 和对外介绍

这一层既要完成品牌统一，也要保留必要的历史上下文，例如：

- “Huppy fork 自 Happy”
- “此前阶段曾命名为 Hoppy”

---

## 五、兼容策略

为了保证“全部改”仍然稳，迁移必须带兼容策略，而不是要求所有旧入口一次性消失。

### 命令兼容

目标命令名是 `huppy`，但在过渡期允许：

- 内部保留 `happy` 或 `hoppy` 的兼容入口
- 帮助文案与截图全部切换为 `huppy`
- 文档明确旧命令只作为兼容别名，不再作为正式品牌使用

### 配置兼容

若已有用户本地配置依赖旧字段或旧路径，应优先考虑：

- 读取时兼容旧值
- 写入时写新值
- 文案中只展示新名字

### 后端兼容

M3 正在准备迁移到自有腾讯云后端，因此这次品牌改名不应重构协议本身。需要坚持：

- 保留现有 Happy 协议语义
- 默认 endpoint 和品牌外壳改为 Huppy
- 不在此次任务中引入额外协议破坏

### 文档兼容

当文档在讨论上游项目或来源项目时，保留 `Happy`：

- “Happy 原版”
- “Happy wire protocol”
- “Happy Server”

当文档在讨论当前产品时，统一改为 `Huppy`。

---

## 六、实施边界

这次任务关注的是**品牌迁移**，不是顺手做大重构。

### 本次要做

- 品牌名和命令名统一设计
- 用户可见文案统一
- 配置和默认入口统一
- 文档和设计稿统一
- 必要的兼容别名设计

### 本次不做

- 后端协议重写
- 目录大搬家导致历史追踪断裂
- 与品牌无关的大规模文件重命名
- 借机清理所有历史 `Happy` 符号
- 因为“名字更干净”而破坏现有运行链路

---

## 七、验证标准

迁移完成后，至少需要满足以下结果：

### 用户视角

- App 内不再出现 `Hoppy`
- 新品牌统一显示为 `Huppy`
- 图标、欢迎页、标题、商店文案一致
- 默认域名为 `huppy.ai`

### 开发者视角

- README、安装方式、命令示例统一使用 `huppy`
- CLI 帮助和通知标题统一为 `Huppy`
- 旧命令如保留，表现为兼容入口而不是主入口

### 技术视角

- App 能正常构建
- CLI 能正常运行
- 默认配置不因改名失效
- 文档中对 Happy 上游来源说明仍然准确

---

## 八、风险与对策

| 风险 | 对策 |
|------|------|
| 命令名全改导致脚本失效 | 先引入兼容别名，再切文档主入口 |
| 粗暴替换误伤 Happy 上游说明 | 替换前按语义分组筛选，不使用单次无差别替换 |
| 包名/slug 改动导致构建或打包异常 | 先改显示名和入口，再逐步验证配置层 |
| 默认 endpoint/域名改动与 M3 冲突 | 把默认值变更和 M3 后端计划对齐，不额外改协议 |
| 文档全改后失去历史上下文 | 在 spec/README 中保留“Happy -> Hoppy -> Huppy”的迁移说明 |

---

## 九、推荐实施顺序

推荐沿用上次 `Happy -> Hoppy` 的成功路径，但范围更完整：

1. 先改品牌入口与配置显示名
2. 再改首页/登录页/欢迎页与图标资源
3. 再改翻译文案与 CLI 用户可见标题
4. 再改命令名、README、默认示例
5. 最后统一文档、spec、plan 和后端默认域名

每一层改完都要做局部验证，再进入下一层。

---

## 十、结论

这次改名不应采取“全局替换一次做完”的办法，而应采用：

**统一目标是全部切到 `Huppy`，执行方式是分层迁移、分层验证、保留必要兼容。**

这样可以同时满足三件事：

- 品牌真正统一到 `huppy.ai`
- 产品继续可用，不把现有链路改坏
- 为后续 M3 腾讯云自托管后端保留稳定基线
